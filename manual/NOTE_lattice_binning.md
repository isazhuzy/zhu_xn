# 为什么分箱图会锯齿 — lattice binning 教训（fig111 MPB 修复，2026-07）

## 一句话
**分箱任何由价格衍生的变量之前，先问"它到底能取哪些值"，把箱中心放在那些值上。**

## 慢速推导

1. **分箱均值图的前提**：相邻箱里装的是"同类"的 tick，这样箱均值才可比。

2. **价格衍生量都长在格点上（lattice）**：
   - 成交价 = tick 的整数倍 → tick 单位下是整数
   - mid = (bid+ask)/2 → 整数或 .5（半格）
   - 两个 mid 的平均 → .0 / .25 / .5 / .75（四分之一格）
   - 所以 MPB = 成交均价 − 两mid均值 只能取 0, ±0.25, ±0.5, … 的格点值。
     几百万行堆在这些点上。

3. **格点之间的值来自另一个种群**：500ms 内成交扫过多个价位（sweep），
   vwap 才会落在格点之间。约 15%，且其后续走势 ~2× 强于单一价位成交。
   （安静盘口一笔成交 ≈ 后续 0.3 tick；扫单 ≈ 0.7 tick。）

4. **BW=0.1 的箱网格对不上 0.25 的格点**：
   - 1.25/0.1 = 12.5，1.75/0.1 = 17.5 —— 正好卡在两箱正中间。
   - `np.round` 是银行家舍入（half-to-EVEN）：12.5→12（bin 1.2），17.5→18（bin 1.8），
     方向不一致。
   - 结果：巨量格点质量落在 1.0/1.2/1.5/1.8/2.0，而 1.1/1.3/1.4/1.6 只装稀有的
     sweep 行。大箱（低均值）与小箱（高均值）沿 x 轴交替 → 连线成规律锯齿。

5. **周期性的抖动 ⇒ 先怀疑测量网格，不是市场。** 随机噪声不会那么有规律。

## 修复（mpb_return_ddb.py）
```python
BW = 0.25                             # 箱宽 = 格点间距，箱中心正好落在格点上
b = np.floor(x / BW + 0.5) * BW       # floor(x+0.5) = 永远 half-up，无奇偶抽签
```
修复后每个箱 = 该格点 + 附近少数 sweep 行的加权混合，相邻箱同质 → 平滑 S 曲线。
回归端（OLS 用原始值）从来没这个问题——纯画图层的伪影，fig111 右图不变即为佐证。
